Map ↔ List cross-linking

Map Navigation
▼
The problem

The Map and Events/Venues tabs are completely siloed. A user who spots a marker on the map has no way to jump to that venue's card in the Venues tab, and a user browsing the Events list has no way to locate the venue on the map. They're forced to manually re-navigate and re-filter.

Suggested fix

Add a "Show on map" button to every event and venue card. Clicking it switches to the Map tab and flies the view to that marker, then opens its popup. Conversely, add a "See events" link in each map popup that switches to the Events tab pre-filtered to that venue.

Implementation sketch

Lift a selectedVenueId state into App.jsx. Pass a setTab/flyTo callback down to the card components. In VenueMap, watch selectedVenueId and call map.flyTo() + open the marker popup via a ref map.

User impactHigh
EffortMedium
FilesApp.jsx, VenueMap.jsx, EventList.jsx, VenueList.jsx
H

Filter state persists across tab switches

Filters
▼
The problem

Filter state (types, regions, datePreset) is held in App.jsx and shared across tabs — which means selecting "Westside" on the Events tab silently restricts the Map and Venues tab too. Worse, some filters (event type, date preset) visually disappear when switching to the Venues tab, leaving users confused about why fewer results appear.

Suggested fix

Split filter state into per-tab buckets: venue-level filters (type, region) remain shared (they're genuinely cross-tab), but event-level filters (event type, date preset, exhibition status) should be scoped to the Events tab only. This also prevents the FilterBar from showing irrelevant controls based on the active tab.

Bonus

Persist filter state to sessionStorage so that refreshing the page (e.g. after following an event link) restores where the user was.

User impactHigh
EffortLow
FilesApp.jsx, FilterBar.jsx
H

Shareable / deep-link URLs

Navigation UX
▼
The problem

All UI state — active tab, mode, filters, selected venue — lives entirely in React state. There is no URL to copy and share. A user who finds a great filter combination (e.g. "Eastside workshops this weekend") can't send it to a friend, and the back button doesn't navigate between views.

Suggested fix

Encode tab, mode, and active filters into the URL hash (e.g. #tab=events&mode=events®ion=eastside&type=workshop&date=weekend). Use a useEffect that writes to window.location.hash on state change, and a one-time initialization effect that reads from it on load. No router library needed for a single-page app of this size.

Implementation sketch
// On mount: read hash → set state // On state change: write hash useEffect(() => { const params = new URLSearchParams(location.hash.slice(1)); if (params.get('tab')) setTab(params.get('tab')); if (params.get('mode')) setMode(params.get('mode')); // …etc }, []); useEffect(() => { const p = new URLSearchParams(); p.set('tab', tab); p.set('mode', mode); if (regions.size) p.set('regions', [...regions].join(',')); // …etc history.replaceState(null, '', '#' + p.toString()); }, [tab, mode, regions, types, …]);

H

Map shows all venues, not just "eventful" ones

Map
▼
The problem

The Map tab currently shows only venues with upcoming one-off events (openings, workshops, etc.), silently hiding the majority of the ~150-venue database. If a user is trying to plan a neighborhood gallery walk, they'll miss venues whose exhibitions don't have a scraper-captured opening reception.

Suggested fix

Add a prominent toggle on the map — "Events only / All venues" — defaulting to "Events only" (current behavior). When switched to "All venues," show the full database with muted markers for venues with no upcoming events. This is consistent with how the Venues tab works, and leverages the existing eventfulFilter boolean already in App.jsx.

This toggle could also sync with the Exhibitions mode: switching to Exhibitions mode could auto-toggle the map to show all venues that have live exhibitions (a different eventfulIds set computed from liveExhibitions).

User impactHigh
EffortLow
FilesApp.jsx, VenueMap.jsx
H

Event cards: add "today" / "tomorrow" / "this weekend" callouts

Events UX
▼
The problem

Event cards display dates as formatted strings like "Thu, May 1, 2026 · 7:00 PM" with no sense of urgency or proximity to now. Users must mentally calculate whether something is happening tonight or in three weeks.

Suggested fix

Add a relative callout label alongside the date: "Today", "Tomorrow", "This weekend", or "Next week" when applicable. These already have logic defined in filters.js (resolveDatePreset), so you can reuse that. Show it as a small highlighted chip next to the date, distinct from the existing type chips.

Also consider

A "Happening now" badge for events where start <= now <= end — especially useful for openings and performances.

User impactHigh
EffortLow
FilesEventList.jsx, filters.js
M

Filter tray is hidden by default — active filters invisible

Filters UX
▼
The problem

The FilterBar collapses by default. When filters are active, the badge count (activeCount > 0) is shown, but users can't see which filters are active without opening the tray. This creates a silent filtering problem: a user who filtered to "Eastside" and then reloads or switches tabs may not realize their results are still restricted.

Suggested fix

When filters are active and the tray is closed, show a summary of active filter chips below the toggle button — small, read-only, dismissible pills showing what's active (e.g. "Eastside · Workshop · This weekend"). Each pill could have an × to remove that single filter without opening the tray. This is a common pattern in marketplace and event apps.

User impactMedium
EffortLow
FilesFilterBar.jsx

Map popups: surface upcoming events inline

Map UX
▼
The problem

Map popups show venue metadata (name, type, neighborhood, description, links) but no event information. A venue marker lit up as "eventful" gives no hint of what's happening or when — the user must switch tabs to find out.

Suggested fix

Pass a filtered view of upcomingEvents to VenueMap and look up events by venue_id inside each popup. Show the next 1–3 events as compact rows (title + date). This requires passing event data into VenueMap, which currently only receives venues.

Events/Exhibitions mode toggle buried in header

Navigation UX
▼
The problem

The Events / Exhibitions mode toggle lives inside the <h1> of the header, making it visually small, hard to discover, and semantically odd (buttons inside a heading). New users are unlikely to find it — especially on mobile where the header wraps tightly.

Suggested fix

Move the mode toggle to be a sub-tab row visible only when the Events tab is active, rendered below or alongside the main tab nav. Alternatively, treat it as a prominent segmented control in the FilterBar area. Either way, it should look like a toggle, not inline text.

Event cards: grouped by date

Events UX
▼
The problem

The Events list sorts by date but presents a flat grid of cards. When viewing "All dates," it's hard to tell at a glance what's happening on a specific day vs. next month — users must read every card's date individually.

Suggested fix

Group events by day with a date section header (e.g. "Thursday, May 1 · 3 events") before each group. This is a natural pattern for event calendars and maps directly onto the existing sorted order. The grouping logic can live in EventList.jsx as a simple reduce over the sorted array.

M

Custom date range picker

Filters
▼
The problem

Date filtering offers only four fixed presets: This weekend, Next week, This month, All dates. There's no way to filter to a specific date range — e.g. "I'm visiting LA May 10–14." The filterEvents function in filters.js already accepts startDate and endDate params, but there's no UI for them.

Suggested fix

Add a "Custom range" option to the date presets. When selected, reveal two native <input type="date"> fields for start and end. Wire them to the existing startDate/endDate params already supported by filterEvents — this is almost entirely a UI addition with minimal logic work.

User impactMedium
EffortLow
FilesFilterBar.jsx, App.jsx
M

Loading state is minimal; no skeleton UI

UX Performance
▼
The problem

While data loads, the app shows only a centered text string: "Loading data…" in a small panel. The entire UI — header, tabs, filter bar — is absent. This looks broken and creates a jarring layout shift when the data arrives.

Suggested fix

Render the Header and FilterBar immediately (even before data loads), and show skeleton cards in place of the event/venue grid. The Leaflet map already has its own loading behavior. Skeletons are pure CSS (grey animated shimmer on placeholder divs) and dramatically improve perceived performance.

User impactMedium
EffortLow
FilesApp.jsx, EventList.jsx, VenueList.jsx

Lazy-load the Leaflet map bundle

Performance
▼
The problem

Leaflet (react-leaflet + leaflet) is a sizeable dependency (~150kb gzipped) that's loaded eagerly on every page visit, even when users land on the Events or Venues tab where the map is never rendered.

Suggested fix

Use React's lazy() + Suspense to dynamically import VenueMap only when the map tab is first activated. In Vite, this is a code-split point: const VenueMap = lazy(() => import('./components/VenueMap.jsx')). This removes Leaflet from the initial bundle entirely.

Expected gain

~150kb off the initial parse cost; faster first paint on mobile networks for non-map users.

User impactLow–Medium
EffortVery Low
FilesApp.jsx
L

Venue detail view / side panel

Data UX
▼
The problem

Venues are surface-level cards with no expandable detail view. A user interested in a specific gallery has to click through to the venue website to learn more — there's no way to see all of a venue's upcoming events, exhibitions, and social links in one place within the app.

Suggested fix

Add a venue detail panel (slide-in drawer or expanded card state) that aggregates: venue description + address, all upcoming events for that venue, currently running exhibitions, and social/website links. This brings together the data that already exists in venues.json and events.json and makes the app feel like a real directory rather than a flat list.

User impactMedium
EffortMedium

L

Scraper coverage visibility for contributors

Data
▼
The problem

The Venues tab shows 150 venues, but contributors have no way to tell which venues have active scrapers vs. which are manually maintained or unscraped. The scrapers/registry.py list doesn't surface anywhere in the UI, making it hard for community contributors to identify gaps.

Suggested fix

Add a "No scraper" chip or indicator to venue cards (and the map popup) for venues that lack an active scraper. The scraped-venues list could be exported as a JSON array at build time (a small Vite plugin or a step in the deploy workflow) and loaded alongside venues.json. This directly supports the contributing workflow described in CONTRIBUTING.md.

User impactLow (contributors)
EffortMedium
Filesscrapers/run_all.py, VenueList.jsx, VenueMap.jsx
